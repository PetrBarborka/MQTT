This page describes evaluation of existing client/server solutions for the MQTT protocol.

Brokers

Brokers come as binary packages that, if run from code, should be probably just put on their own process and left alone. It is generally not expected for the Broker to need to be modified to application specific needs, as they get addressed on the Client side.

HiveMQ

  • Nonfree, price not specified as of yet
  • Windows/Unix
  • Binary broker, works, probed locally with mqtt-spy on OSX

Mosquitto

  • Free, open source, massive support
  • Windows/Unix/Anything that can build C++
  • Binary broker, works, probed locally with mqtt-spy on OSX

Erlang MQTT daemon

  • Free, open source, commercial support
  • Windows/Unix/Anything that can build Erlang
  • Binary broker, works, probed locally with mqtt-spy on OSX

Verne

  • Pretty much the same as emqttd
  • Free, open source, commercial support
  • Windows/Unix/Anything that can build Erlang
  • DIDN'T BUILD on OSX

Whitesoft

Czech commercial broker, promises to send demo over email, looks like a poor mans HiveMQ. Meh.

Mosca

A broker on Node.js. Didn't try.

IBM

IBMs efforts on both client and server side either mutated into cloud services, or merged into mosquitto.

Seems like mosquitto is a clear winner - free, open-source, supported and doing what all the others are doing.

Clients

Clients come as libraries that are expected to be implemented to suit application specfic needs.

Paho

  • C, C++, C#, embeds(metal C?) and many other bindings
  • Free, opensource

Libmosquitto

  • C with C++ ∧ Python bindings
  • Free, opensource
  • Appears to be only partially documented
  • powers Facebook messaging

IBM

merged into libmosquitto

There are many other client libraries, but they are usually either not fully developed (only QoS0 for example) or constrained in some other way (targeting exclusively embed, written in weird language ... ). I will give both Paho and libmosquitto a try.
